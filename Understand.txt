📚 CONCEPT 3: Bcrypt (Password Hashing)
═══════════════════════════════════════════════════════════════════════════════

WHAT IT IS:
- Bcrypt: A special algorithm that converts passwords into secret codes

WHY WE NEED IT:
- NEVER store passwords in plain text (very dangerous!)
- If hacker steals database, they can't read passwords
- Even the server admin can't see your actual password

WHERE I IMPLEMENTED IT:
  📁 src/models/User.js (Line 35-46)

HOW I DID IT:
  ```
  Before saving to database:
  password: "myPassword123" → 
  bcrypt hashes it →
  Stored as: $2b$10$k9x8f7g6h5j4... (random gibberish)
  
  When user logs in:
  They type: "myPassword123"
  Bcrypt compares it to stored hash
  If match → Login successful
  If no match → Login failed
  ```

SECURITY LEVEL:
  - Even if hacker gets the database
  - They see: $2b$10$k9x8f7g6h5j4...
  - Can't reverse it to find original password
  - ✓ Your password is safe


📚 CONCEPT 4: JWT Tokens (Authentication)
═══════════════════════════════════════════════════════════════════════════════

WHAT IT IS:
- JWT = JSON Web Token
- A secret token that proves "you are logged in"
- Like a movie ticket - shows you're allowed entry

WHY WE NEED IT:
- After login, user gets a token
- User sends token with every request
- Server checks: "Is this token valid?"
- If yes → allow access, if no → block access

WHERE I IMPLEMENTED IT:
  📁 src/controllers/authController.js (Token generation - Line 5-10)
  📁 src/middleware/auth.js (Token verification - 35 lines)

HOW I DID IT:

  STEP 1 - Generate Token After Login:
  ```
  User logs in successfully
  → Server creates: jwt.sign({id: user._id}, SECRET_KEY)
  → Returns: "eyJhbGciOiJIUzI1NiIsInR5cCI..." (long string)
  → User saves to browser/app
  ```
  
  STEP 2 - Use Token in Requests:
  ```
  User wants to access protected route
  → Sends: Authorization: Bearer eyJhbGciOiJIUzI1NiI...
  ```
  
  STEP 3 - Verify Token:
  ```
  Server receives token
  → jwt.verify(token, SECRET_KEY)
  → If valid → Allow access
  → If invalid/expired → Block access (401 error)
  ```

TOKEN EXPIRATION:
  - Tokens expire after 7 days (set in .env)
  - After expiration, user must login again
  - This is for security - limits damage if token is stolen

EXAMPLE:
  Login: curl -X POST http://localhost:8000/api/auth/login
  Response: {"token": "eyJhbGciOiJIUzI1NiIsInR5cCI..."}
  
  Then use token: curl http://localhost:8000/api/auth/me \
    -H "Authorization: Bearer eyJhbGciOiJIUzI1NiI..."

📚 CONCEPT 6: Middleware (Auth Protection)
═══════════════════════════════════════════════════════════════════════════════

WHAT IT IS:
- Code that runs BEFORE handling a request
- Like security guards at a building
- Checks: "Are you allowed to enter?"

WHERE I IMPLEMENTED IT:
  📁 src/middleware/auth.js (35 lines)

HOW I DID IT:
  ```
  Request comes → auth.middleware runs:
  
  1. Check: Do they have Authorization header?
     No → Block (401 error)
  
  2. Extract token from header: "Bearer TOKEN"
  
  3. Run jwt.verify(token, SECRET_KEY)
  
  4. If valid → Add user data to request, allow forward
     If invalid → Return error
  ```

USAGE IN ROUTES:
  ```
  // Protected route
  router.get('/me', authMiddleware, getCurrentUser);
  
  // Flow:
  Request → authMiddleware checks token → If OK → getCurrentUser runs
  ```

WHO CAN ACCESS IT:
  ✓ Users with valid JWT token
  ✗ Anyone without token
  ✗ Anyone with expired token
  ✗ Anyone with fake token

📊 DATA FLOW FOR PHASE 1
═══════════════════════════════════════════════════════════════════════════════

REGISTRATION FLOW:
User fills signup form
    ↓
POST /api/auth/register with username, email, password
    ↓ (authController.register)
Validate inputs → Check email not taken
    ↓ (models/User.js)
Hash password with bcrypt
    ↓
Save to MongoDB
    ↓
Generate JWT token
    ↓
Return token + user data to client
    ↓
Client saves token to localStorage/AsyncStorage


LOGIN FLOW:
User enters email & password
    ↓
POST /api/auth/login
    ↓ (authController.login)
Find user by email in MongoDB
    ↓
Compare entered password with stored hash (bcrypt)
    ↓
If match → Generate JWT token
If no match → Return 401 error
    ↓
Return token + user data to client


PROTECTED REQUEST FLOW:
Client sends: GET /api/auth/me with "Authorization: Bearer TOKEN"
    ↓
Request enters auth.js middleware
    ↓
Extract token, verify with JWT
    ↓
If valid → Extract user ID from token, add to request
If invalid → Return 401, block further execution
    ↓
If valid, next handler runs (authController.getCurrentUser)
    ↓
Fetch user from MongoDB
    ↓
Return user data
