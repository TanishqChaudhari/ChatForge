🔄 FINAL SYSTEM FLOW (What Project Demonstrates)
═══════════════════════════════════════════════════════════════════════════════

COMPLETE USER JOURNEY:

1. User Registration
   User fills signup → POST /api/auth/register → 
   Bcrypt hashes password → Saved to MongoDB → JWT generated → 
   Token returned to client

2. User Login
   User enters credentials → POST /api/auth/login → 
   Email found in MongoDB → Bcrypt compares password → 
   Match = true → JWT generated → Token returned

3. User Connects to Chat (Phase 2+)
   User has JWT token → WebSocket connect with JWT auth →
   Redis marks user as online → Sent to all connected users

4. User Sends Message (Phase 2+)
   User types message → Click send → 
   Message stored in MongoDB → Kafka event produced →
   Socket.io sends message to recipient(s) in real-time →
   Message marked as "delivered" → Recipient sees it

5. User Reads Message (Phase 2+)
   Recipient clicks/opens message → 
   Status updated to "read" in MongoDB →
   Socket.io sends read receipt back to sender →
   Sender sees "read at [timestamp]"

6. User Online Status (Phase 2+)
   User connects → Redis: {userId: online} →
   All users notified user is online
   
   User disconnects → Redis: remove userId →
   Socket.io broadcasts user offline
   
   Check online users → GET /online-users → Fetch from Redis

7. Message Queue Processing (Phase 2+)
   Message saved to DB → Kafka event produced →
   Consumer receives event → Logs to analytics →
   Triggers notifications → Updates cache

8. User Leaves Chat
   User closes app → Socket.io disconnect →
   Redis removes user → Broadcasted to all →
   MongoDB stores last activity timestamp
