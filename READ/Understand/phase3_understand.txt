📚 CONCEPT 1: What are WebSockets?
═══════════════════════════════════════════════════════════════════════════════

TRADITIONAL HTTP (POLLING):
  Client: "Any new messages?" 
    ↓ (Request)
  Server: "No"
    ↓ (Response - Connection closes)
  Client waits 2 seconds...
  Client: "Any new messages?"
    ↓ (Request again)
  Server: "No"
    ↓ (Response - Connection closes)
  [Repeat every 2-5 seconds]
  
  Inefficient! ❌ Wastes bandwidth, high latency


WEBSOCKETS (PERSISTENT CONNECTION):
  Client connects once → Server keeps connection open
    ↓ (Full-duplex connection)
  Server: [User goes online] → Instantly push update to all connected clients
  Client receives within milliseconds ✓ Real-time!
  
  No polling needed! ✓ Server can push anytime


SOCKET.IO (WEBSOCKET WRAPPER):
  - Provides WebSocket fallback for older browsers
  - Reconnection handling automatic
  - Event-based messaging (easier than raw WebSockets)
  - Broadcasting to multiple clients
  - Room management


✓ WHAT WE USE: Socket.io for real-time in ChatForge


📚 CONCEPT 2: Socket.io Authentication with JWT
═══════════════════════════════════════════════════════════════════════════════

PROBLEM: How to verify user identity over WebSocket?
  - HTTP has Authorization header, WebSockets don't
  - Need to authenticate BEFORE connection handshake

SOLUTION: JWT Token in Socket.io Auth

Code:
  ```javascript
  // Server: Middleware runs BEFORE connection
  io.use((socket, next) => {
    const token = socket.handshake.auth.token;
    const decoded = jwt.verify(token, SECRET);
    socket.userId = decoded.id;  // Attach to socket
    next();  // Allow connection
  });
  
  // Client: Send token during connection
  const socket = io('http://localhost:8000', {
    auth: {
      token: myJWTToken  // Send token here
    }
  });
  ```

USER JOURNEY:
  1. Frontend: User logs in via REST /api/auth/login
  2. Get back: JWT token
  3. Frontend: Connect Socket.io with token in auth
  4. Server: Middleware verifies token
  5. Server: Can now trust socket.userId
  6. Connection established ✓


IDENTITY FLOW:
  socket.userId = "alice_id"
    ↓
  socket.join("alice_id")  // Personal room
    ↓
  Message from Bob to Alice
    ↓
  io.to("alice_id").emit('message_received', {message})
    ↓
  Only Alice receives it ✓ (not everyone!)


📚 CONCEPT 3: Real-time Message Delivery
═══════════════════════════════════════════════════════════════════════════════

WHAT HAPPENS WHEN ALICE SENDS MESSAGE TO BOB:

1. Alice clicks Send
2. Frontend emits: socket.emit('send_message', {conversationId, content})
3. Server receives event
4. Server saves message to MongoDB:
   ```
   {
     senderId: alice_id,
     content: "Hi Bob",
     conversationId: "conv_123",
     status: "sent"
   }
   ```
5. Server sends confirmation back to Alice:
   socket.emit('message_sent', {messageId, status: 'delivered'})
   → Alice's app shows: ✓ (checkmark)

6. Server sends to Bob (if online):
   io.to(bob_id).emit('message_received', {message})
   → Bob's app instantly shows: "Alice: Hi Bob"

7. Update conversation.lastMessage → Shows in chat list


WHAT IF BOB IS OFFLINE?
  - Message saved to MongoDB (step 4)
  - When Bob connects next time
  - GET /api/conversations/:id/messages retrieves all messages
  - App shows: "Alice: Hi Bob" from history
  - ✅ No messages lost!


TIMING:
  Text typed → Sent button clicked → Server receives
    ↓ (Network latency: ~50-200ms)
  Saved to DB → Confirmation sent to Alice
    ↓ (50-200ms)
  Total: ~100-400ms from click to "sent" ✓ Instant!
  
  Compare HTTP polling: Would be 2000-5000ms


📚 CONCEPT 4: Room Management
═══════════════════════════════════════════════════════════════════════════════

WHAT IS A ROOM?
  - A way to group socket connections
  - Think: A broadcast group
  - Server sends message → All in room receive it
  - Multiple sockets can be in same room

SOCKET.IO ROOMS:

1. Personal Room (user's ID):
   ```javascript
   socket.join(userId);  // Alice joins "alice_id" room
   io.to("alice_id").emit('event', data)  // Only Alice receives
   ```
   
   Use: Private messages to specific user
   Example: io.to(alice_id).emit('message_received', message)

2. Conversation Room:
   ```javascript
   socket.join(conversationId);  // Join "conv_123" room
   io.to("conv_123").emit('event', data)  // All in conversation receive
   ```
   
   Use: Typing indicators, read receipts
   Example: io.to(convId).emit('user_typing', {user: 'Alice'})

3. Broadcast (all connected):
   ```javascript
   io.emit('event', data)  // Everyone receives
   ```
   
   Use: Server announcements
   Example: io.emit('server_maintenance', {startTime: ...})


📚 CONCEPT 5: Typing Indicators
═══════════════════════════════════════════════════════════════════════════════

WHAT IT IS:
  White dots appearing in WhatsApp/Messenger when someone types
  Visual feedback that other person is composing


HOW IT WORKS:

Alice:
  - Starts typing in text input
  - Keyboard event fires
  - Emit: socket.emit('typing_start', {conversationId})
  - Server broadcasts to conversation room
  
Bob receives:
  - socket.on('user_typing', (data) => ...)
  - Show UI: "Alice is typing..."
  
Alice:
  - Finishes typing or clicks Send
  - Emit: socket.emit('typing_stop', {conversationId})
  - Server broadcasts
  
Bob receives:
  - socket.on('user_stopped_typing', (data) => ...)
  - Hide UI: Remove "typing..." indicator


IMPLEMENTATION:
  ```javascript
  // Server
  socket.on('typing_start', (data) => {
    io.to(data.conversationId).emit('user_typing', {
      userId: socket.userId,
      username: socket.user.username
    });
  });
  
  // Client (Bob)
  socket.on('user_typing', (data) => {
    showTypingIndicator(data.username);
  });
  ```


WHY THIS EXISTS:
  - Real conversations have pauses while typing
  - Without this: Looks like conversation is dead
  - With this: Feel of real-time conversation


📚 CONCEPT 6: Read Receipts (Message Status)
═══════════════════════════════════════════════════════════════════════════════

MESSAGE LIFECYCLE WITH RECEIPTS:

Alice sends: "Hi Bob!"
  ↓
Message status: "sent" ✓ (stored in DB)
  ↓
Transmitted to Bob's device → status: "delivered" ✓✓
  ↓
Bob opens conversation and sees message
  ↓
App marks as read → status: "read" ✓✓✓
  ↓
Send receipt back to Alice
  ↓
Alice sees ✓✓✓ (fully read)


HOW IT'S STORED:

In MongoDB:
  ```
  {
    _id: "msg_123",
    content: "Hi Bob",
    senderId: alice_id,
    conversationId: "conv_123",
    status: "read",              // updated multiple times
    deliveredAt: "2026-03-26T22:15:00Z",
    readAt: "2026-03-26T22:15:30Z"
  }
  ```


SOCKET.IO FLOW:

Bob's app (when Conv opened):
  socket.emit('message_read', {messageId, conversationId})
  ↓
Server:
  - Find message in DB
  - Update: status = "read", readAt = new Date()
  - Save to DB
  ↓
Server notifies Alice:
  io.to(senderId).emit('message_read_receipt', {messageId})
  ↓
Alice's app:
  - Update message in UI from ✓✓ to ✓✓✓
  - Show "Read [timestamp]" under message (optional)


FRONTEND BEHAVIOR:
  - ✓ = Message sent to server successfully
  - ✓✓ = Message confirmed delivered to device
  - ✓✓✓ = User has read (conversation opened)
  
  Like: WhatsApp, Telegram, Messenger


📚 CONCEPT 7: Online/Offline Presence
═══════════════════════════════════════════════════════════════════════════════

TRACKING USER STATUS:

On Connection:
  - Alice logs in and connects Socket
  - Server: activeUsers['alice_id'] = socket_id
  - Broadcast: io.emit('user_online', {userId: 'alice_id'})
  - Alice appears as green dot in contacts list

On Disconnection:
  - Alice closes app or loses connection
  - Server: delete activeUsers['alice_id']
  - Broadcast: io.emit('user_offline', {userId: 'alice_id'})
  - Alice's green dot disappears

Frontend Usage:
  - Show green dot next to user name if online
  - Show "Last seen 5 min ago" if offline
  - Chat list sorting: Online users first


CODE:
  ```javascript
  io.on('connection', (socket) => {
    const userId = socket.userId;
    activeUsers[userId] = socket.id;
    
    socket.broadcast.emit('user_online', {userId});
    
    socket.on('disconnect', () => {
      delete activeUsers[userId];
      socket.broadcast.emit('user_offline', {userId});
    });
  });
  ```


📚 CONCEPT 8: Error Handling in WebSocket
═══════════════════════════════════════════════════════════════════════════════

DIFFERENT FROM HTTP:

HTTP Error:
  POST /api/messages
    ↓
  Response: 400 Bad Request ← Client checks status code


Socket.io Error:
  socket.emit('send_message', {data})
    ↓
  Server checks validation
    ↓
  If error: socket.emit('error', {message: 'Invalid data'})
    ↓
  Client: socket.on('error', (error) => {...})


COMMON SOCKET.IO ERRORS:

Authentication Errors:
  ❌ No token: "Authentication required"
  ❌ Invalid token: "Invalid token"
  Result: Connection rejected before handshake

Operation Errors:
  ❌ Not participant: "You are not a participant"
  ❌ Conv not found: "Conversation not found"
  ❌ Missing fields: "Missing required fields"
  Result: socket.emit('error', {message})


CLIENT HANDLING:
  ```javascript
  socket.on('error', (error) => {
    console.error('Socket error:', error);
    showToast(error.message);
  });
  
  socket.on('connect_error', (error) => {
    console.error('Connection error:', error);
    // Reconnect automatically or show UI
  });
  ```


📚 CONCEPT 9: Socket.io vs REST API
═══════════════════════════════════════════════════════════════════════════════

WHEN TO USE SOCKET.IO (Real-time):
  ✓ Sending instant messages
  ✓ Typing indicators
  ✓ Read receipts
  ✓ User presence
  ✓ Live collaboration
  ✓ Push notifications
  → latency < 100ms

WHEN TO USE REST API (Request-Response):
  ✓ Getting message history
  ✓ Fetching user profiles
  ✓ One-time operations
  ✓ Mobile data-saving mode
  ✓ Fallback for unreliable connections
  ✓ Server-to-third-party integrations
  → Can tolerate higher latency


HYBRID APPROACH (ChatForge):
  - Send message? Use Socket.io (instant feedback)
  - Get history? Use REST API (pagination)
  - Need confirmation? Can use either
  - Offline? Fall back to REST API
  - Both save to same database ✓

📚 CONCEPT 11: Message Persistence
═══════════════════════════════════════════════════════════════════════════════

WHEN USER OFFLINE:

Sent messages:
  - Saved to MongoDB immediately
  - If sender closes app: restored from DB on login
  ✓ No message loss

Received messages:
  - Saved when sender emits
  - If Bob offline: message waits in DB
  - When Bob reconnects: GET /messages retrieves history
  ✓ No message loss

Typing indicator:
  - Real-time only
  - No persistence needed
  - If Alice offline while typing: no indicator shown ✓

Read receipt:
  - Updated in DB when Bob marks read
  - Persisted for history
  ✓ Alice can see "Read at 10:30pm" weeks later


DATABASE QUERIES:
  ```
  When Bob reconnects: GET /conversations/conv_id/messages
  Query: Message.find({conversationId, ...})
                 .sort({createdAt: -1})
                 .limit(20)
  Returns: All messages including old ones
  ```
