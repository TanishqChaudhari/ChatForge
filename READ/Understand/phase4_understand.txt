🎓 WHAT PROBLEM DO WE SOLVE?
═══════════════════════════════════════════════════════════════════════════════

Problem 1: "Who's Online Right Now?"
  ❌ Without Redis: We have activeUsers = {} in memory
     Problem: If server restarts, we lose all user data
     Problem: Can't scale to multiple servers
  
  ✅ With Redis: All online users stored in persistent cache
     Solution: Survives server restart
     Solution: Can scale to many servers (all read same Redis)


Problem 2: "I Need to Process Messages Asynchronously"
  ❌ Without Kafka: Message sent → only users in memory can see it
     Problem: What if you want to log, analyze, or notify externally?
     Problem: Message events are lost after handling
  
  ✅ With Kafka: Message sent → event stored in topic
     Solution: Multiple consumers can subscribe (logging, notifications, analytics)
     Solution: Events persist for replay and recovery
     Solution: Loose coupling between services

## 🔑 KEY CONCEPT 1: REDIS FOR PRESENCE
═══════════════════════════════════════════════════════════════════════════════

What is Redis?
  - **In-memory data store** (like a super-fast database)
  - Perfect for: Caching, sessions, real-time data
  - Speed: 1000x faster than MongoDB for simple key-value lookups
  - Trade-off: Data lost if server crashes (usually OK for presence)

Simple Redis Analogy:
  Think: A whiteboard in an office
  
  MongoDB = Filing cabinet (permanent, secure, slow)
  Redis = Whiteboard (temporary, fast, visible to everyone)


PRESENCE TRACKING WITH REDIS:

```
User Connects          User in App          User Closes App
    ↓                      ↓                       ↓
SET user:alice:online   GET user:alice:online  DEL user:alice:online
   true                  → true                 (deleted)
   EX 300                                       
   (5 min)
```

When you ask "Who's online?":
  1. Get all keys: KEYS user:*:online
  2. Foreach user_id in results:
     - Fetch user details from MongoDB
     - Add to "Online Users" list
  3. Return list in < 10ms (super fast!)


WHY EX 300 (5 MINUTE EXPIRY)?

Scenario A: User loses internet connection
  - App crashes unexpectedly
  - Socket.on('disconnect') might not fire
  - Redis key expires after 5 minutes
  - User automatically marked offline
  
Scenario B: User properly disconnects
  - socket.on('disconnect') fires immediately
  - DELETE user:alice:online
  - User marked offline instantly

## 📨 KEY CONCEPT 2: KAFKA FOR EVENTS
═══════════════════════════════════════════════════════════════════════════════

What is Kafka?
  - **Distributed event streaming platform**
  - Think: A recording device for all events that happen
  - Perfect for: Event sourcing, audit logs, real-time processing
  - Guarantee: No event is lost

Simple Kafka Analogy:
  Think: A newspaper printing system
  
  - Writer (Producer): Writes message → sends to newspaper office
  - Newspaper office (Kafka): Records the message
  - Reader (Consumer): Can read any message anytime
  - Archive: Messages stored forever (or configurable time)


KAFKA FLOW:

Producer sends event:
  ```
  socket.emit('send_message', {...})
    ↓
  Message saved to MongoDB
    ↓
  produce_to_kafka('messages' topic):
    {
      messageId: 'msg_123',
      senderId: 'alice',
      content: 'Hello Bob',
      timestamp: '2026-03-27T10:30:00Z'
    }
  ```

Kafka stores event:
  ```
  Topic: 'messages'
  Partition 0:
    [0] {messageId: 'msg_1', ...}
    [1] {messageId: 'msg_2', ...}
    [2] {messageId: 'msg_3', ...}
    ← Never deleted (unless policy says to)
  ```

Consumers subscribe:
  ```
  Consumer 1 (Analytics): Reads all -> stores stats
  Consumer 2 (Logging): Reads all -> writes to file
  Consumer 3 (Notifications): Reads all -> sends email/SMS
  
  All read the SAME events without interfering!
  ```


WHY KAFKA IS POWERFUL:

1. **Decoupling** (Services don't know about each other)
   Without Kafka:
   - Message sent → Must immediately call notification service
   - Message sent → Must immediately call analytics service
   - Message sent → Must immediately call logging service
   - If any service fails → Whole system broken
   
   With Kafka:
   - Message sent → Produce event to Kafka
   - Each service subscribes independently
   - Services can fail/restart without affecting others

2. **Replay** (Can process history)
   Without Kafka:
   - Message events are processed and discarded
   - No way to analyze past messages
   
   With Kafka:
   - All events stored permanently
   - New service can "catch up" by reading past events
   - Can replay if you change business logic

3. **Multiple Consumers** (Many services listening)
   Without Kafka:
   - Only one place knows about events
   
   With Kafka:
   - Logging service listens
   - Analytics service listens
   - Notification service listens
   - Email service listens
   - They all see the same events!

## 🏗️ ARCHITECTURE: HOW REDIS + KAFKA WORK TOGETHER
═══════════════════════════════════════════════════════════════════════════════

```
PRESENCE TRACKING (Redis):
  Session → Socket.io → Redis
  Who's online right now
  
  Example: show green dot next to "Alice" in user list
  
  Flow:
  1. socket.on('connect') → SET user:alice:online true EX 300
  2. GET /api/users/online → KEYS user:*:online → fetch from MongoDB
  3. Return: [{id, username, email}, ...]


EVENT STREAMING (Kafka):
  Message → Producer → Kafka Topic → Consumers
  What happened (history of all messages)
  
  Example: Log all messages to analytics database
  
  Flow:
  1. socket.emit('send_message') or POST /api/messages
  2. Save to MongoDB
  3. Produce to Kafka topic 'messages'
  4. Consumer reads event
  5. Log to analytics / send notification / etc


VISUALIZED:

Frontend App
  │
  ├─→ Socket.io Connection
  │     │
  │     └─→ send_message event
  │          │
  │          ├─→ Save to MongoDB
  │          │
  │          ├─→ SET Redis (presence) ← Real-time "who's online"
  │          │
  │          └─→ Produce to Kafka ← Event for other services
  │
  │
  ├─→ GET /api/users/online
  │     │
  │     └─→ KEYS user:*:online (from Redis)
  │          │
  │          └─→ Fetch details from MongoDB
  │              Return: List of online users
  │
  │
  └─→ GET /api/messages/:convId
        │
        └─→ Query MongoDB (message history)
            Return: All past messages
```

## 🔄 REAL-WORLD EXAMPLE: USER COMES ONLINE
═══════════════════════════════════════════════════════════════════════════════

STEP 1: User opens ChatForge app
  Browser → connects Socket.io

STEP 2: Socket.io handshake
  JWT verified, socket created, userId = "alice_id"

STEP 3: Connection handler executes
  ```
  socket.on('connect') → 
    1. activeUsers['alice_id'] = socket_id (memory)
    2. Redis: SET user:alice_id:online true EX 300
    3. Broadcast: emit('user_online', {userId: 'alice_id'})
  ```

STEP 4: What happens now?

  In Redis: key "user:alice_id:online" exists
  
  If someone calls GET /api/users/online:
    → KEYS user:*:online → finds "user:alice_id:online"
    → extracts alice_id
    → Query MongoDB for user alice_id
    → Returns: {id: 'alice_id', username: 'Alice', email: 'alice@...', online: true}

  In broadcast:
    → All other connected sockets see: "Alice is online"
    → Their UI updates: green dot appears next to Alice

STEP 5: User sends message
  Alice → socket.emit('send_message', {content: 'Hi Bob'})
  
  Server:
    1. Save to MongoDB (Message collection)
    2. Emit to Bob: user_online event
    3. Produce to Kafka:
       {
         event_type: 'message_sent',
         messageId: 'msg_123',
         senderId: 'alice_id',
         content: 'Hi Bob',
         timestamp: '...'
       }
  
  Kafka topic 'messages' now has this event
  Any consumer listening will get notified

STEP 6: User closes app
  Browser closes socket connection
  
  socket.on('disconnect') →
    1. delete activeUsers['alice_id']
    2. Redis: DEL user:alice_id:online
    3. Broadcast: emit('user_offline', {userId: 'alice_id'})

  If someone calls GET /api/users/online now:
    → KEYS user:*:online → "user:alice_id:online" NOT FOUND
    → Alice not in results
    → Her UI shows grey dot (offline)

## 💾 REDIS vs MONGODB: WHEN TO USE WHICH
═══════════════════════════════════════════════════════════════════════════════

REDIS: Fast, Temporary
  ✅ User presence (who's online)
  ✅ Session tokens
  ✅ Cache of frequently accessed data
  ✅ Real-time counters
  ❌ NOT: Permanent records
  ❌ NOT: Complex queries
  ❌ NOT: Transactions

MONGODB: Durable, Complex
  ✅ Messages (need permanent record)
  ✅ Users (need complex queries)
  ✅ Conversations (relationships)
  ✓ Complex filters and sorting
  ❌ NOT: Real-time fast lookups (slower than Redis)


CHATFORGE USAGE:

  MongoDB:
    - Messages (permanent record of every message)
    - Users (permanent user accounts)
    - Conversations (permanent conversation history)
    
  Redis:
    - User online status (temporary, expires)
    - Session caching (temporary)
    
  Kafka:
    - Event log (what happened: message sent, user online, etc.)

## 🚀 SCALING BENEFIT: MULTIPLE SERVERS
═══════════════════════════════════════════════════════════════════════════════

WITHOUT REDIS (Single Server):
```
Server 1:
  activeUsers = {
    alice_id: socket_123,
    bob_id: socket_456
  }

User opens app → connects to Server 1 ✓
Friend asks "Who's online?" → queries Server 1 ✓

Problem: If Server 1 crashes?
  All online data LOST
  activeUsers = {} (empty)
```


WITH REDIS (Multiple Servers):
```
Server 1                    Redis                Server 2
activeUsers = {       ←→  user:alice:online  ←→  activeUsers = {
  alice_id: s1_123         user:bob:online          bob_id: s2_456
}                      }

User Alice connects to Server 1
  → activeUsers['alice_id'] = s1_123 (Server 1 memory)
  → Redis: SET user:alice:online true
  
User Bob connects to Server 2
  → activeUsers['bob_id'] = s2_456 (Server 2 memory)
  → Redis: SET user:bob:online true

Someone asks Server 2: "Who's online?"
  → Server 2 checks Redis (not its own memory)
  → Finds BOTH alice AND bob
  → Returns: Alice and Bob are online ✓

If Server 1 crashes:
  Alice's connection lost but Redis still has user:alice:online
  (expires after 300 seconds)
```

BENEFIT: **Can scale horizontally!**
  - Add Server 3, Server 4, etc.
  - All check same Redis
  - All see same online users
  - Single point of truth

## 📊 KAFKA: PRODUCER & CONSUMER PATTERN
═══════════════════════════════════════════════════════════════════════════════

PRODUCER (Our ChatForge):
  "I'm sending a message, let everyone know about it"
  
  ```
  produceMessageEvent(message) {
    1. Convert message to JSON
    2. Connect to Kafka broker
    3. Send to topic 'messages'
    4. Confirm sent
  }
  ```

CONSUMER (Anyone who cares):
  "I want to know about all messages"
  
  ```
  subscribeToMessages() {
    1. Connect to Kafka broker
    2. Subscribe to topic 'messages'
    3. Wait for events
    4. When event arrives:
       - Log it
       - Send notification
       - Update analytics
       - Archive it
       - etc.
  }
  ```

MULTIPLE CONSUMERS EXAMPLE:

  Topic 'messages': [msg_1, msg_2, msg_3, msg_4, ...]
  
  Consumer 1 (Logging):
    Reads: msg_1 → log to file
    Reads: msg_2 → log to file
    Reads: msg_3 → log to file
    
  Consumer 2 (Analytics):
    Reads: msg_1 → count by user
    Reads: msg_2 → count by user
    Reads: msg_3 → count by user
    
  Consumer 3 (Notifications):
    Reads: msg_1 → send email to recipient
    Reads: msg_2 → send email to recipient
    Reads: msg_3 → send email to recipient

  All 3 consume the same messages without interfering!

## 🔐 SECURITY CONSIDERATIONS
═══════════════════════════════════════════════════════════════════════════════

REDIS:
  ⚠️ By default, NO password authentication
  ✓ Solution: Enable --requirepass in Redis config
  
  ⚠️ Data in memory, not encrypted
  ✓ Solution: Use Redis over SSL/TLS
  
  ⚠️ Anyone with network access can read/write
  ✓ Solution: Firewall Redis to private network only


KAFKA:
  ⚠️ By default, messages not encrypted
  ✓ Solution: Enable SSL/TLS for brokers
  
  ⚠️ No authentication by default
  ✓ Solution: Enable SASL (Simple Authentication & Security Layer)
  
  ⚠️ All consumers see all messages
  ✓ Solution: Use topic-level ACLs to restrict access


PRODUCTION CHECKLIST:
  ✓ Redis with password authentication
  ✓ Redis on private network
  ✓ Kafka with SSL/TLS enabled
  ✓ Kafka with SASL enabled
  ✓ Firewall rules configured
  ✓ Regular backups of critical data
